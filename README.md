## 売り上げ予測AI
- 企業の売り上げを企業HPのテキストから予測します。
- URL: https://haru-first-app-19443304909.asia-northeast1.run.app/docs


| クラス | 売上レンジ |
|--------|------------|
| S | 10兆円〜 |
| A | 1兆〜10兆円 |
| B | 5000億〜1兆円 |
| C | 〜5000億円 |

## 特徴量エンジニアリング
- **テキスト埋め込みベクトル**
    - テキストエンコーダーモデル：`multilingual-e5-base`
    - 処理フロー: ページ種別ごとのエンコード→ベクトルの結合
    - 意図と仮説
        - 企業のグローバル化による英語表現の増加：英語表現がテキストないで重要なトークンであったときに日本語のモデル`bert-base-japanese-v3`においては逆にノイズになってしまうのではないか。
        - 事前学習方法の違い：BERTなどのMLMは、事前学習の手法がマスクしたトークンを当てる自己教師あり学習にのみ基づいていることから、テキストの意味・文脈を埋め込むことに適していると考えられるが、今回のような分類タスクにおいてはファインチューニングが前提である。一方で、E5モデルの事前学習方法は、弱教師あり学習による対照学習に基づいており、事業領域・方針など売り上げ予測において重要な要素を意味空間上でしっかりと区別することができるのではないか。
        - `multilingual-e5-base`の評価結果を示すテクニカルレポートにおける日本語の検索能力の評価は非常に高く、多言語だからといって日本語への精度が劣るわけではないから[@wang2024multilinguale5textembeddings]。
       
- **追加特徴量**
    - 詳細を`/notebook/eda.ipynb`に記載(キーワード特徴量・TF-IDF特徴量)
## モデル設計
- **モデル概要**
    - 入力：テキストベクトル＋正規表現によるキーワード特徴＋構造特徴量
    - 出力：売り上げクラス

    - ネットワーク構成

    | セクション | 処理内容 |
    | :--- | :--- |
    | テキストMLP | Dense(256) → BN → Dropout(0.3) → Dense(64) → BN → Dropout(0.3) |
     キーワード | Dense(64) → BN → Dropout(0.2) |
    |結合層 | 両MLPの出力を結合|
    | 共通MLP | Dense(256) → BN → Dropout(0.4) → Dense(classes) |

    - 活性化関数:
        - 隠れ層：`ReLU` 
        - 出力層：`Softmax`
    - 過学習防止のため、各層に BatcNormalization と Dropout を適用した
- **設計意図**
    - 入力をテキスト埋め込みベクトル＋キーワード・構造特徴にしたのは`Silhouette Score`が優れていたEDAの結果を踏またから。
    - それぞれの特徴に対して全結合層を挟んでから結合を行ったのは次元数などのモダリティの違いからいきなり結合すると学習が壊れるのではないかと判断したため。
    - テキストベクトルのみを用いたTF-IDFを使ったゲーティング処理によるテキストベクトルでも同程度のいい結果が得られたが(`/model/model_tuning.ipynb`)、推論精度のブレイクスルーが見られなかったことと、トークン辞書作成はサンプルデータへの依存性が高く、今回サンプルデータに含まなかった世界の企業やベンチャー企業と辞書との不一致を踏まえAPIとしての機能性を考慮し採用しなかった。

## 振り返り
- **精度の考察**
  - 今回のテストで興味深かったのが、日産やSUBARUなどの日本を代表する自動車企業がCクラスに誤分類されているということだ。同じ上位クラスでかなりの確信度でAクラスに分類された三菱商事株式会社や株式会社セブン＆アイ・ホールディングスと比較してみると、自動車会社特有のものづくりに関する技術的なテキストがかなりあることが分かった。ものづくりという観点で見ると、Cクラスにはメーカなどの企業は各クラスに多数ある一方で、総合商社のようなビジネスモデルに関する特別なアイデンティティを持つ企業は下位クラスにはあまり存在しない。このように自動車業界が広い枠組みで下位クラスの企業と重なることが原因で推定が難航したのではないか。
  - 同じテレビ業界の株式会社フジ・メディア・ホールディングスと日本テレビホールディングス株式会社は正解クラスと推定クラスで上下関係が逆転していた。レスポンスjsonを見ると、クローリングした１つのページに対しての取得文字数に圧倒的な差があることがわかる。かなりの確信度でAクラスに誤分類され、１ページあたり約160文字と情報量が少ない日本テレビの推定はテキストベクトルによる特徴表現がうまくなされず、特定のキーワード特徴によるハイリスクな推論がなされたのではないか。また、他の企業と比べて圧倒的な文字数を得られたフジテレビは実際にサイトをみてみると、不祥事に関する記述が多く見受けられノイズとして捉えられたのではないかと考えた。このように文字数が少ないことによる局所的な特徴抽出や文字数が多いことによるノイズの発生を回避することがクローリングの際に求められる要件であると考えられる。例えば、クローリングするページの深さや、文字数の最適化などだ。
- **MLOps観点での考察**
- 本プロジェクトでAPIを実装した際に、以下の２つの問題が発生した。
  1. 株式会社フジ・メディア・ホールディングスのURLを入力した時だけランタイムエラー(300秒)が発生する：テスト結果の`text_length_total`を見ると、他の会社と比較して圧倒的にクローリングした文字数が多いことが分かる。この問題を解決するためにクローリングに7000文字の文字数制限を施した。これによりランタイムエラーは回避されたが、データの量とのトレードオフな関係があり、レスポンス速度かデータの特徴表現の多さどちらを優先するべきかを目的によって考える必要があるのではないか。また、サイトの構造的は企業によってさまざまであり、今回はトップページにいるという前提でクローリングを試みたが、実際に企業サイトの構造を分析してクローリング手法の最適化を図る必要があると思った。例えば、トップページからダイレクトにIRページなどののURLが存在するだろうとこちらで勝手に仮定していることなど。
  2. 時間が経ってから、コールドスタート状態でリクエストを送信すると、モデルのファイルダウンロードによってランタイムエラー(300秒)が発生する：本APIは課金制限のためにCloud Runの最小インスタンス数を0に設定している。そのため、コールドスタート時にはモデルファイル等のダウンロードが発生していた。売り上げ予測モデル自体のデータ容量は17MBくらいだったので原因はおそらくHugging Faceの事前学習モデルのダウンロードにあると考えられる。そこで、Cloud Buildでアップロードする際のDockerイメージをビルドする段階で事前学習モデルをダウンロードしておく必要があった。これにより、巨大ファイルのダウンロードは避けることができるがDockerイメージの容量は巨大化するというデメリットがある。また、CI/CD的にはアジャイル開発におけるソースコードの軽微な変更のたびに巨大なDockerコンテナを何度もビルドする必要がある。そのため、巨大なファイルはクラウド上のストレージ等に保管することが推奨されるのだと思った。
